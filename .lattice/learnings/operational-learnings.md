# Operational Learnings

Actionable patterns harvested from practice. Loaded at the start of design/implementation sessions; tightened over time.

## Design

- **Zero-services CRUD is valid** (2026-06-11, Properties & Rooms design): Under Rails MVC+, a feature whose writes all stay inside one aggregate legitimately has no service objects. Check the anemic-service anti-pattern in `architecture.md` before scaffolding services; controllers calling models directly is the correct shape for single-aggregate CRUD.
- **Guard mass-assigned values that map to a DB constraint at every HTTP boundary** (2026-06-11 enums; tightened 2026-06-15, Maintenance Requests review for FKs): A mass-assigned value backed by a database constraint 500s on bad input unless guarded. Two flavors seen: (1) string **enums** raise `ArgumentError` on unknown values (no validation error) — needs a rescue→422 at every action, not just the obvious status one; (2) **foreign keys** (e.g. `assignee_id`) for a non-existent record hit the DB FK constraint → `ActiveRecord::InvalidForeignKey` (500) — guard with a model existence validation (`validate :assignee_must_exist, if: -> { assignee_id.present? }`) so it flows through the controller's `RecordInvalid` rescue to 422. The form's `<select>` is never the trust boundary.
- **State machine events generate methods — check for column clashes at contract time** (2026-06-11, Guests & Reservations design): AASM events like `check_in` define `check_in`/`check_in!` methods that silently collide with same-named attribute readers. When a lifecycle verb matches a column concept, suffix date/time columns (`check_in_on`, `checked_out_at`) and let events keep the clean verbs. Catch this at Level 4 (Contracts), not during implementation.
- **Name states for what they assert** (2026-06-11, Properties & Rooms design): Choose domain vocabulary that won't collide with future derived concepts. A room is `operational`, not `available` — availability is derived from reservations later. When naming an enum value, ask what adjacent features will need the obvious word for.
- **Audit existing writers when a feature mutates shared aggregate state** (2026-06-15, Maintenance Requests design): When a new feature writes to a column another aggregate's service already writes, audit those existing services at contract time for silent overrides/ordering conflicts. `MaintenanceRequest` and `Reservations::CheckOut` both set `Room.status`; an unguarded `CheckOut` would flip `out_of_service → cleaning`, dropping an active maintenance block and making a broken room bookable. Caught at Level 4, fixed with a guard (`change_status!("cleaning") unless active maintenance`). Trigger: any feature that writes a column it didn't introduce — grep the column's other writers before finalizing contracts.
- **Audit/activity logs are aggregate children, stored structured** (2026-06-15, Reservation Detail design): When a feature needs a unified audit/activity feed, model it as a **child of the aggregate root** (`has_many ... dependent: :destroy`, reached only through the root), not a standalone or polymorphic table. Store the payload as **structured `details` (JSON)** — never pre-rendered English — so the no-hardcoded-strings I18n rule holds and the view localizes at render. Centralize entry construction on the root (one `log_*`/`add_*` method) so the same logging isn't re-implemented across each write service; the acting user is passed in as `actor:` (models never read `Current`). Lifecycle transitions that are single-aggregate stay model methods that append the entry in a transaction; only cross-aggregate writes need a service (architecture.md §8).

## Security

- **Reject `/\` as well as `//` when validating user-supplied redirect/return paths** (2026-06-15, New guest during booking review): An open-redirect guard that only checks `start_with?("/") && !start_with?("//")` is bypassable: browsers normalise the backslash in `/\evil.com` to `/`, turning it into a protocol-relative URL to evil.com. Rails' default `redirect_to allow_other_host: false` does *not* catch it — Ruby parses `/\evil.com` as a relative URI with `host=nil`, so it reads as same-origin. Validate with `%r{\A/[^/\\]}` (a leading slash followed by a non-slash, non-backslash char). Keep such path-safety logic out of `app/helpers` (view-only per architecture.md) — put it on ApplicationController and expose to views via `helper_method` so controllers and views share one guard.

## Implementation

- **Reuse the read-side scope as the write-side guard** (2026-06-13, Guests & Reservations review): When a business rule is already encoded as a query scope for listing (e.g. `Room.available_between`), the service that performs the corresponding write should call that same scope to guard, not re-derive the rule inline. `BookRoom` had re-implemented "not out_of_service + no overlap" as two separate checks, duplicating the scope's logic — two places to keep in sync. Per ddd §6, availability lives in one scope; the guard becomes `raise unless Room.available_between(period).exists?(id: room.id)`. **Extension** (2026-06-15, Reservation Detail design): a "modify" path (reschedule, change room) enforces the *same* invariants as the original create — reuse the availability scope and the past-date guard rather than re-deriving. The only difference from booking is self-overlap: add an `except:` to the read scope (`Room.available_between(period, except: reservation)`) so a stay doesn't conflict with its own current hold.
- **Style every state value, not just today's** (2026-06-13, Guests & Reservations review): A `status-#{value}` CSS convention silently degrades when a new model reuses it with new values — reservation pills (`status-booked`, etc.) rendered colorless because only the room statuses had color variants. When introducing a second consumer of a per-value class convention, add the new variants in the same change.
- **Generate-and-validate identifiers belong in `before_validation`, not `before_create`** (2026-06-14, Reservation Reference Codes): When a model assigns its own value that is also validated — e.g. a generated `internal_id` code with `presence`/`uniqueness` — use `before_validation on: :create`. Validations run *before* `before_create`, so a `before_create` assignment fails the presence check on every create (caught only when `BookRoom`/factories hit it). Make the callback idempotent (`return if value.present?`) so caller- or factory-supplied values are respected and only blanks are generated. Decide the hook at contract time (Level 4), not when tests fail.
- **Re-verify blueprint host files against current HEAD before wiring** (2026-06-15, Maintenance Requests implementation): A blueprint names specific host actions/files (e.g. "badge on `properties#show`"), but merges landing between design and implementation can relocate them — the reservations-console feature had turned `properties#show` into a redirect and moved the room dashboard to `rooms#index`. Grep for the named host before editing; the design *intent* (badge on the room dashboard) survives even when the address changes. Flag the relocation as a deviation rather than silently following a stale contract.
- **When folding an index into a parent show, watch for the same set being queried twice** (2026-06-13, Property Ops Overview review): A read/dashboard action that assembles several grouped collections can quietly query one record set in two shapes. `properties#show` loaded checked-in reservations both as `@in_house` (a list) and `@current_reservations` (indexed by room for occupancy) — identical rows, two queries. Derive one from the other in Ruby (`@current_reservations = @in_house.index_by(&:room_id)`) instead of re-querying. Also unifies the access idiom (`@property.reservations` throughout, not a mix with `Reservation.where(room:)`).
