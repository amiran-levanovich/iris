# Maintenance Requests

> Track maintenance issues per room through a lifecycle (open → in_progress → resolved, or cancelled). An open request takes the room out of service so it cannot be booked; clearing the last open request restores it. Requests are optionally assigned to a staff member. Managed from the room page; open-request counts surface on the property ops cockpit. This is the third core-domain area named in the knowledge base (properties→rooms, guests→reservations, maintenance requests), and the only one previously unbuilt.

## Constraints

<!-- Non-negotiable once recorded. Add only when confirmed. -->

- Builds on merged Properties & Rooms (Room aggregate, `status` enum incl. `out_of_service`) and the Property Ops cockpit. New table + model expected (schema change, unlike the cockpit redesign).
- Rails MVC+ (architecture.md): the cross-aggregate room-status effect lives in a verb-named service inside a transaction, never in an AR callback (ddd §5 — no domain events / callback side-effects in v1). Single-aggregate writes (create/edit a request, lifecycle transitions that don't touch the room) may go controller→model directly.
- AASM owns the `status` column directly (no Rails enum on the same column) — same pattern as `Reservation`; status state scopes generated by AASM.
- Availability stays derived: `Room.available_between` already excludes `out_of_service`, so once the service sets that status, no booking-path change is needed.
- All user-facing strings via I18n (lazy `t(".key")`, enum labels via `human_enum`). Plain CSS tokens; reuse/extend the `status-<value>` convention (add maintenance state variants in the same change — operational-learnings: "style every state value").

## Design: Level 1 -- Capabilities

Approved 2026-06-15.

1. **Log a maintenance issue against a room** — staff create a request from the room page with title/description, category, priority. Starts `open`.
2. **Opening a request takes the room out of service** — room becomes unbookable; `Room.available_between` already excludes `out_of_service`, so booking is blocked automatically.
3. **Move a request through its lifecycle** — `start` (open→in_progress), `resolve` (→resolved), `cancel` (open/in_progress→cancelled). Illegal transitions rejected gracefully.
4. **Closing the last active request restores the room** — when no open/in_progress request remains, room returns to `operational`.
5. **Optionally assign a request to a staff member** — set/clear assignee (User) on create or edit.
6. **See a room's maintenance history** — list of requests (active + closed) on the room page.
7. **Spot maintenance on the cockpit** — `properties#show` room rows show a ⚠ open-request count.

Out of scope (v1): property-wide maintenance board, per-request blocking flag (every active request blocks), cost/parts tracking, recurring maintenance, notifications/email, attachments, remembering a non-maintenance prior room status on restore.

## Design: Level 2 -- Components

Approved 2026-06-15.

### Aggregate model (DDD)

`MaintenanceRequest` is a **new aggregate root** (entity), not a child of Room — independent lifecycle. Matches the row reserved in ddd-principles.md §1. `belongs_to :room` and `belongs_to :assignee` (User, optional) — cross-aggregate refs by id. Room is never mutated via a callback on the request (ddd §5). No new value objects: `category`/`priority` are string enums; `title`/`description` are strings (ddd §3, don't wrap behavior-free columns).

AASM: `open → in_progress → resolved`; `cancel` from open/in_progress. "Active" = open or in_progress (holds the room).

### Decisions (Level 2)

- One `CloseRequest` service (outcome param) rather than separate Resolve/Cancel — both share the identical "release room if no other active request" effect; splitting duplicates it. Resolve and cancel are one use case (stop tracking → free room) with different outcomes.
- `start!` (open→in_progress) and `update` (reassign/edit) are single-aggregate writes with no room effect → controller calls the model directly (no service; avoids anemic-service anti-pattern, architecture §7).
- Room-nested maintenance `index` doubles as the per-room maintenance page (rooms have no `show` action).

## Design: Level 3 -- Interactions

Approved 2026-06-15. "Active" = open or in_progress.

- **Open (create)** — `POST /rooms/:room_id/maintenance_requests` → `Maintenance::OpenRequest.new(room:, attributes:, assignee:).call`; TXN: create request (`:open`) + `room.change_status!(:out_of_service)`. Success → redirect `room_maintenance_requests_path` + notice; invalid → 422 re-render `:new` (txn rolled back, room untouched); unknown enum → rescue `ArgumentError` → 422. No booking-path change — `Room.available_between` already excludes `out_of_service`.
- **Start** — `PATCH /maintenance_requests/:id/start` (shallow member) → `request.start!` (open→in_progress); room stays out_of_service. Single-aggregate, model direct. Illegal → `rescue_from AASM::InvalidTransition` → redirect + flash alert.
- **Resolve / Cancel** — `PATCH /maintenance_requests/:id/{resolve|cancel}` → `Maintenance::CloseRequest.new(request:, outcome:).call`; TXN: fire AASM event, then `room.change_status!(:operational)` **iff** `MaintenanceRequest.active_for(room).none?`. The "others still active?" check is a query on the MaintenanceRequest aggregate inside the service, not a Room→MaintenanceRequest reach (ddd §1).
- **Reassign / edit** — `PATCH /maintenance_requests/:id` updates title/description/priority/assignee; status unchanged → no room effect → model direct.
- **Cockpit read** — `properties#show` loads active counts grouped by room in one query (`MaintenanceRequest.active.where(room: @property.rooms).group(:room_id).count`); `rooms/_room` renders ⚠ badge linking to the room's list when count > 0.

### Edge cases (v1)

- Maintenance on an occupied room is allowed — out_of_service blocks new bookings only; an existing checked_in reservation is unaffected.
- Room stays out_of_service until the *last* active request closes (`active_for(room).none?` guard).
- Opening a second request on an already-out_of_service room is an idempotent no-op status write.

### Decisions (Level 3)

- Restore target on last-close = always `operational` (see resolved open question).
- Lifecycle transitions split by room-effect: `start`/`update` model-direct (no room write); `create`/`resolve`/`cancel` via service (room write inside txn).

## Design: Level 4 -- Contracts

Approved 2026-06-15.

### Table `maintenance_requests`

| Column | Type | Constraints |
|---|---|---|
| `room_id` | bigint | null: false, FK→rooms, indexed |
| `assignee_id` | bigint | null: true, FK→users, indexed |
| `title` | string | null: false |
| `description` | text | nullable |
| `category` | string | null: false (enum) |
| `priority` | string | null: false, default `"medium"` (enum) |
| `status` | string | null: false, default `"open"` (AASM) |
| timestamps | | |

Indexes: `[:room_id, :status]` (serves `active_for(room)` + cockpit `group(:room_id)` count), `:assignee_id`.

### Model `MaintenanceRequest`

- `belongs_to :room`; `belongs_to :assignee, class_name: "User", optional: true`.
- `enum :category, { plumbing, electrical, hvac, appliance, furniture, structural, other }` (string-valued); `enum :priority, { low, medium, high, urgent }`.
- `validates :title, presence: true`.
- `scope :active, -> { where(status: %w[ open in_progress ]) }`; `scope :active_for, ->(room) { active.where(room: room) }`.
- AASM on `:status` (no Rails enum on that column): states `open`(initial)/`in_progress`/`resolved`/`cancelled`; events `start` (open→in_progress), `resolve` (open|in_progress→resolved), `cancel` (open|in_progress→cancelled).
- **Clash check (operational-learning):** `start!/resolve!/cancel!` + state scopes don't collide with any column; AASM `open` vs `Kernel#open` is harmless. ✔

### Room (add) — `has_many :maintenance_requests, dependent: :destroy` (rooms have no destroy route).

### Services (`app/services/maintenance/`)

- `Maintenance::OpenRequest` — `.call`/`new(room:, attributes:)`; TXN: `room.maintenance_requests.create!(**attributes)` (raises `RecordInvalid` → controller) + `room.change_status!("out_of_service")`; returns request.
- `Maintenance::CloseRequest` — `new(request:, outcome:)` (`outcome ∈ %i[resolve cancel]`); TXN: `request.public_send("#{outcome}!")` (raises `AASM::InvalidTransition` → controller) + `request.room.change_status!("operational") if MaintenanceRequest.active_for(request.room).none?`; returns request.
- `Reservations::CheckOut` **patched**: `room.change_status!("cleaning") unless MaintenanceRequest.active_for(room).any?` (keep out_of_service when the room has active maintenance).

### Controller `MaintenanceRequestsController`

- `set_room` (index/new/create), `set_request` (edit/update/start/resolve/cancel).
- `rescue_from AASM::InvalidTransition` → redirect `room_maintenance_requests_path(@request.room)` + alert.
- `create` → `OpenRequest.call`; `rescue RecordInvalid` (use `error.record`) / `rescue ArgumentError` (bad enum) → 422 `:new`.
- `start` → `@request.start!`; `resolve`/`cancel` → `CloseRequest.call(request:, outcome:)`.
- `update` → `@request.update!`; same 422 rescues.
- Strong params: `params.expect(maintenance_request: %i[ title description category priority assignee_id ])`.

### Routes — `maintenance_requests` nested under `rooms` (shallow), `only: %i[index new create edit update]` + member `patch :start/:resolve/:cancel`.

### `PropertiesController#show` — `@maintenance_counts = MaintenanceRequest.active.where(room: @property.rooms).group(:room_id).count`; `rooms/_room` badge → `room_maintenance_requests_path(room)` when count > 0.

### i18n & CSS

- `maintenance_requests.{index,new,edit,form,actions}.*`, `.{create,start,resolve,cancel}.notice`, `.invalid_transition`; enum labels under `activerecord.attributes.maintenance_request.{categories,priorities,statuses}.*` (via `human_enum`); `properties.show.maintenance_count`.
- CSS: `.status-{open,in_progress,resolved,cancelled}` pill variants + warning badge — add all four state colors in one change (operational-learning "style every state value").

### Specs

- Model: title validation; legal/illegal AASM transitions; `active`/`active_for`.
- Service: `OpenRequest` creates open + room out_of_service; `CloseRequest` restores room iff last active, stays blocked otherwise; `CheckOut` keeps out_of_service when active maintenance exists.
- Request: create blocks room; start/resolve/cancel; invalid transition → redirect+alert; `properties#show` badge count.
- Factory: `maintenance_requests`.

## Design Summary

**Status: Approved -- ready for implementation** (2026-06-15)

- **Components/layers**: new `MaintenanceRequest` model + migration (Models); `Maintenance::OpenRequest` + `Maintenance::CloseRequest` (Services); patched `Reservations::CheckOut`; new `MaintenanceRequestsController` + `PropertiesController#show` count (Controllers); `maintenance_requests` views + `rooms/_room` badge (Views); routes/i18n/CSS; factory + model/service/request specs.

- **Architectural constraints**: cross-aggregate room-status writes live in transactional services, never AASM callbacks (ddd §5); single-aggregate writes (`start!`, `update`) go controller→model; enums guarded at the HTTP boundary; availability stays derived (`available_between` already excludes out_of_service — no booking-path change); all strings via I18n; plain CSS tokens.
- **Domain model**: `MaintenanceRequest` is its own aggregate root referencing Room + User (assignee) by id; "active" = open|in_progress holds the room; no new value objects; no domain events.
- **Open questions resolved**: restore target = always `operational`; one `CloseRequest` (outcome param) not two; `CheckOut` guarded against dropping an active maintenance block.

### Deviations from blueprint (intentional, both improvements)

1. **Cockpit badge host: `rooms#index` (Housekeeping tab), not `properties#show`.** Since the blueprint was written, the reservations-console feature turned `properties#show` into a redirect and moved the room dashboard to `rooms#index`. The badge lives where the room table now renders — same component, current address.
2. **`OpenRequest.call(room:, request:)` instead of `(room:, attributes:)`.** The controller builds the request and handles enum-boundary safety (mirroring `RoomsController#assign_room_attributes`), so the service receives a domain object, not params — a cleaner fit for the architecture boundary rule (services take extracted values/AR objects, never params).
